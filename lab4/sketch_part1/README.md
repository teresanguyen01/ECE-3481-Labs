### Part 1: Concurrency

Concurrency refers to a program’s ability to manage multiple tasks so that they appear to run simultaneously. On an Arduino, however, there is only a single processor core, so only one process can run at a time. Instead of executing processes in parallel, the system rapidly switches between them. When this switching happens quickly enough, it creates the illusion that multiple processes are running at the same time. On systems with multiple cores, separate processes can truly run in parallel on different cores. In this lab, we simulate concurrency by creating multiple processes, each with its own stack. A scheduler switches between these processes using yield() and a timer interrupt so that each process gets a turn to execute.

According to the specification, the function process_select(unsigned int cursp) is called with interrupts disabled. The argument cursp represents the stack pointer of the currently running process. If there is no current process, or if the current process has terminated, cursp will be 0. The function should return 0 if no processes are ready to run, or return the stack pointer of the next process that should run.

The scheduler works as follows. If cursp != 0, the current process is still active, so its stack pointer is saved and the process is placed back into the ready queue so that it can run again later. If cursp == 0, this indicates that the process has terminated or does not exist, so it is not requeued. After handling the current process, the scheduler checks the ready queue. If the queue is empty, there are no processes available to run, so current_process is set to NULL and the function returns 0. If the queue contains processes, the next process is removed from the queue using dequeue(), set as the current_process, and its stack pointer is returned so that execution can resume from that process.

To support scheduling, we implement a ready queue using a FIFO (first-in, first-out) linked list. This queue stores process structures representing each runnable process. The queue maintains both a head pointer and a tail pointer so that processes can be efficiently added and removed. Initially, both head and tail are set to NULL.

The queue requires two operations: enqueue and dequeue. During the enqueue operation, we first check whether the queue is empty. If it is empty, the new process becomes both the head and the tail. Otherwise, the current tail’s next pointer is updated to point to the new process, and the tail pointer is moved to the new process. During the dequeue operation, the process at the head of the queue is removed and returned. The head pointer is updated to the next process in the list. If removing the head causes the queue to become empty, the tail pointer is also set to NULL.

Each process is represented by a struct process_state, which contains a stack pointer (sp) and a pointer (next) linking to the next process in the queue. The global variable current_process keeps track of the currently running process. To create a process, the function process_create(void (*f)(void), int n) allocates a new process_t, initializes its stack using process_init(f, n), and inserts the process into the ready queue. The function returns -1 if process creation fails and 0 on success.

The function process_start() begins concurrent execution by setting current_process to NULL and calling process_begin(). Context switching occurs through the timer interrupt system. The timer periodically calls yield(), which triggers process_timer_interrupt(). This interrupt saves the current processor state, calls process_select() to determine the next process to run, restores the state of that process, and resumes execution. Through this mechanism, the system continuously switches between processes, creating the appearance that multiple tasks are running simultaneously.
